# ADR-0001: Architecture review (2026-04-22)

## Context

Solo-developer personal Bible study + journal app (React 19 + TS + Vite, Supabase backend, Google-Drive migration in PLAN Phase 3). PLAN.md is the rulebook; R1–R13 in `~/.claude/CLAUDE.md` are the hard rules. This review is a cold second-session pass over `master` at `06d9cc2` (post PR #4 edge-function R3 fix, post PR #6 test audit, post PR #10 lasso-move fix). Headline stats: 27 files over R4's 300-LOC budget, 861 vitest tests running in 3.23s, 4 Playwright e2e specs (958 LOC), Supabase egress at 97 % of free tier, NotabilityEditor at 3,632 LOC (PLAN says 2,700 — it has grown during the audit).

## Assessment by principle

**Modularity — MIXED.** `services/` is well sliced per concern (auth, sync, each AI provider, each storage domain). `hooks/` is thin and well-named. The problem lives in `components/`: four top-of-codebase files hold most of the UI logic — `NotabilityEditor.tsx` (3,632 LOC), `JournalView.tsx` (2,405), `ChatInterface.tsx` (1,691), `EnhancedNotebook.tsx` (1,610). These are god components: render + input + state + persistence in one file. Evidence: `components/NotabilityEditor.tsx:1–3,632` holds types, markdown conversion (`markdownToHtml` at L177, `maybeConvertMarkdown` at L305, `looksLikeMarkdown` at L329), slash-command definitions, stroke rendering, text-box editing, AI connector lines, lasso, page management, and the toolbar — 111 `useState/useRef/useEffect` call sites and 87 handler callbacks in one closure.

**Contract APIs — WEAK.** Modules mostly communicate through function exports plus ambient state in `localStorage` / IndexedDB. There is no declared "public vs internal" boundary — `services/aiProvider.ts:32` reads `localStorage` directly from any caller; `services/syncService.ts:62–75` owns `bible-app-sync-state` with no exposed getter/setter contract; `App.tsx:335` does `await import('./services/supabase')` mid-function to reach `supabase`, `authManager`, `canSync()` as globals. There is one good counter-example: `services/systemPrompts.ts` exports `BIBLE_SCHOLAR_SYSTEM_PROMPT`, `AI_LANGUAGE_DIRECTIVE`, `SPLIT_MARKER` as typed constants with a dedicated R3 tripwire test (`services/__tests__/systemPrompts.singleSource.test.ts`). That is the pattern to generalize.

**Extensibility — MIXED.** Adding an AI provider today is painful: edit `services/aiProvider.ts:20` (union type), `aiProvider.ts:30` (`ALL_PROVIDERS`), `aiProvider.ts:257` (`providerNames` map — second copy), `aiProvider.ts:295` (`OPENAI_COMPATIBLE_ENDPOINTS` — third copy), plus seven parallel maps in `supabase/functions/ai-chat/index.ts:21–102` (`PROVIDER_ENDPOINTS`, `PROVIDER_KEY_NAMES`, `DEFAULT_MODELS`, `ALL_MODELS`, `VISION_MODELS`, `PROVIDER_NAMES`, `ANTHROPIC_PROTOCOL_PROVIDERS`). Fifteen providers × ~10 maps = every new provider is a 10-site edit (R3 smell, cross-cutting). Adding a paper type or slash command, by contrast, is a one-liner — `SLASH_COMMANDS` at `NotabilityEditor.tsx:66` is a clean data-driven extension point. Adding a sync backend today requires rewriting `syncService.ts` wholesale; PLAN Phase 3 acknowledges this.

**Readability / maintainability — MIXED.** Good: naming is consistent (`handle*` for event handlers, storage services named `*Storage`, services named by subject noun). Comments explain *why* (see the history block in `services/systemPrompts.ts:9–19`). Bad: no `@ts-ignore`, but file size is out of control — 27 files over the 300-LOC R4 budget; mean size of the top 14 UI components is ~1,400 LOC. `components/BibleViewer.tsx.bak` and `components/ChatInterface.tsx.backup` are orphan files that R7 says should be deleted on sight. Three in-repo `TODO_JOURNAL_*.md` files at repo root are another R7 smell.

**Testing strategy — WEAK.** `TEST-AUDIT.md` (2026-04-21) is the single best artifact in this repo; it honestly concludes that the 861-test corpus produced false confidence — SPLIT, sync, pencil, and egress regressions all shipped past it. The ratio of unit-shaped mock-shuffle tests to integration e2e tests is roughly 861:4. The ~861 vitest tests total ~32 files of assertions against mock return values (`services/__tests__/webSearch.test.ts` has 9.0 mocks/test; `journalRealtimeSync.test.ts`, `syncService.test.ts`, `autoSaveResearchService.test.ts` are near-fully mocked per TEST-AUDIT Table C). The bright spots: `chineseConverter.test.ts`, `chatBibleReferences.test.ts`, `systemPrompts.singleSource.test.ts`, and the Playwright `tests/e2e/notability-pointer-matrix.spec.ts` (593 LOC, the pointer matrix PLAN demanded). There is no clear fast-unit / slow-integration stratification — `npm run test:slow` does not exist yet, though TEST-AUDIT recommends it.

**Error handling (R5) — WEAK.** 215 `try/catch` sites across 30 files; 80 of them catch without rethrowing or surfacing. Grep for `// silently` returns 31 hits; none produce user-visible toasts, only comments. Worst offenders are `components/BibleViewer.tsx` (6 silent catches, e.g. L322, L328, L463, L599, L608, L621, L684), `hooks/useBibleDownload.ts` (6), `services/annotationStorage.ts:93,140,163`, `App.tsx:174,198,239,276,353` (TODO-tagged), `services/syncService.ts:1133,1148` (full-sync step failures swallowed silently — this is how egress and sync bugs ship without the user noticing). `components/ErrorBoundary.tsx:23–26` logs nothing — `componentDidCatch` has `// TODO: use error reporting service` with a `void errorInfo` suppression, so a crash hits the fallback UI with no record kept. R5 compliance is aspirational, not achieved.

**Data flow — MIXED.** Ownership is IndexedDB → UI reads, Supabase → sync. The sync layer is single-threaded via a mutex (`services/syncService.ts:96–109`) with exponential backoff (L112–133), and incremental syncs fetch server timestamps first (L168–181) — that part is well-engineered. Race conditions live at the UI/IndexedDB boundary: `App.tsx:243–278` (`handleSaveNote`) reads `notes` state, mutates IndexedDB, then re-reads — if two autosaves overlap the second can stamp stale content. `syncService.ts:222–233` "only overwrite local if remote is newer than our last sync" uses `syncState.lastNotesSync` as the local timestamp, not the per-note timestamp, so a local edit made *before* the last sync pull can be silently clobbered on next pull. This is last-writer-wins at the sync level but not at the item level.

**Performance & scale — MIXED.** Good: lazy-loaded route components at `App.tsx:18–31`, code splitting by panel, IndexedDB for Bible text so the majority of reads never hit network. Bad: `vendor-opencc` bundle is 1.1 MB (opencc-js in `package.json:30`), loaded even on first paint if the user switches Chinese variants; `@tiptap/*` bundle is four packages (`package.json:22–26`) though only rich-text editing needs it. Canvas render: `services/strokeNormalizer.ts:renderAllStrokes` runs on every pointer-move in `NotabilityEditor.tsx` — no dirty-region invalidation. Sync egress: the journal list fetches `notability_data` (base64 image blobs embedded) in the list query — PLAN Phase 2 calls this out correctly; `syncService.ts:197–264` is the file that needs lazy-column-selection.

## Top 10 actionable recommendations

1. **Split `NotabilityEditor.tsx` now, before any more features.** PLAN's 5-module split (Canvas / TextBox / AI / SlashMenu / shell) is the right shape, but start narrower: extract `markdownToHtml`, `maybeConvertMarkdown`, `looksLikeMarkdown` (L177–341) into `services/markdownConvert.ts` with unit tests; then lift `SLASH_COMMANDS` + slash executor into `components/notability/SlashMenu.tsx`; then extract text-box toolbar. Attempting the full 5-way split in one session will violate R2 (one session, one concern). Scope: **L** (multi-session).

2. **Extract a `Provider` interface and collapse the 10 parallel maps.** In `services/aiProvider.ts` and `supabase/functions/ai-chat/index.ts`, define `interface Provider { name, label, endpoint, keySetting, defaultModel, allModels, visionModels, protocol: 'openai'|'anthropic'|'gemini' }` and ship one `PROVIDERS: Provider[]` array. Adding the 16th provider becomes one record. Scope: **M**.

3. **Delete the two `.backup` / `.bak` orphans on sight (R7).** `components/BibleViewer.tsx.bak`, `components/ChatInterface.tsx.backup`. Git history is the backup. Scope: **XS**.

4. **Audit and fix the 31 `// silently handle` sites (R5).** Each needs to either (a) surface a user-visible toast via the existing `setToast` pattern, (b) rethrow with context, or (c) get a concrete one-line reason (not "silently handle"). Start with `App.tsx:174,198,239,276,353` — those are top-level user-visible failures. Scope: **M**.

5. **Ship the TEST-AUDIT demotion plan.** Create `npm run test:slow`, rename the Table-C offenders to `*.slow.test.ts` (start with `webSearch`, `journalRealtimeSync`, `syncService`, `autoSaveResearchService`). Then run the regression-replay exercise TEST-AUDIT queues. Target ≤150 fast tests. Scope: **S** (demotion) + **M** (replay).

6. **Fix the per-item sync conflict window in `syncService.ts:197–264`.** Compare per-note `updated_at` vs a local `notes_modified_at` map rather than a single `lastNotesSync` timestamp. This eliminates the silent local-loss case. Scope: **S**.

7. **Add a `bottom-right status pill` for sync state (PLAN Phase 1).** With R5 cleanup in (4), surface sync errors visibly. Ship it with a Playwright test that forces offline and asserts the pill state. Scope: **S**.

8. **Lazy-column the journal list query.** `services/syncService.ts` journal functions — `select id,title,updated_at,location_name,verse_ref` only on list view; fetch `notability_data`/`blocks` only when an entry opens. Direct mitigation of 97 % egress. Scope: **S**.

9. **Extract stroke rendering and history into `services/notabilityState.ts`** (plain reducer). Separating the ~100-state-slot reducer from `NotabilityEditor.tsx` unlocks unit-testing (currently the editor has zero coverage) and is a precondition for the bigger split in (1). Scope: **M**.

10. **Define `services/ai/Provider.ts` as the seam for Phase 3 browser-side racing.** Both today's edge function and tomorrow's browser racer will consume the same `Provider` record shape. Do this now — before `googleDriveSyncService.ts` — so Phase 3 doesn't compound the duplication. Scope: **S**.

## Anti-patterns already present

- **God components.** `NotabilityEditor.tsx` (3,632), `JournalView.tsx` (2,405), `ChatInterface.tsx` (1,691), `EnhancedNotebook.tsx` (1,610). All exceed R4 by 5×–12×.
- **Ambient-state communication.** `aiProvider.ts:33,40,55,65` all read/write `localStorage` directly; no injected config object. Every consumer must know the key names.
- **Import-side-effect for service init.** `App.tsx:14` — `import './services/syncService'` runs the module top-level; the module subscribes to auth changes as a side effect. Untestable and order-sensitive.
- **Parallel provider maps.** `ai-chat/index.ts:21,40,50,61,79,96` — six maps keyed on the provider string. Adding one provider = six coordinated edits. Exactly the pattern R3 exists to prevent.
- **Swallowed `catch {}`.** `syncService.ts:1133` ("Individual step failure or timeout doesn't block others") — no telemetry, no toast; the user sees a green "synced" state while data silently didn't sync. This is likely contributing to the egress mystery: failed syncs never surface, so users retry, inflating traffic.
- **`// TODO: use error reporting service` scattered 8×** — a known gap that's been present long enough to get a standard comment. Ship Sentry or just a `reportError(e, ctx)` wrapper around `console.error` so these stop being silent.
- **Mock-shuffle tests.** Per TEST-AUDIT Section C — tests whose mocks *are* the subject. Running these 29s (actually 3.23s; fast) on every commit gave the codebase a green-CI habit while real bugs shipped.
- **Repo-root TODO markdown files.** `TODO_JOURNAL_PRINT.md`, `TODO_JOURNAL_SYNC.md`, `TODO_JOURNAL_UX.md`, `APPLE_PENCIL_IMPROVEMENTS.md`, `FEATURE_SUMMARY.md`, `GENERAL_RESEARCH_VIEWER_SUMMARY.md` — notes that belong in PLAN.md or `docs/`, not at repo root.

## What's done well

- **The process itself.** R1–R13 + the R13 automated pre-push review + TEST-AUDIT + this ADR slot — this is unusual discipline for a solo project and it is working: PR #4 closed the edge-function R3 loophole (`supabase/functions/ai-chat/index.ts:13,631` now imports `BIBLE_SCHOLAR_SYSTEM_PROMPT`).
- **`services/systemPrompts.ts` and its tripwire test.** The module has the history embedded in its own docstring (L9–19) — future-Claude will not re-introduce the 5-copy SPLIT bug because `systemPrompts.singleSource.test.ts` will fail. This is what every cross-cutting constant in the repo should look like.
- **Sync mutex and exponential backoff** (`syncService.ts:96–138`). Correctly structured, even if individual step failures are too quiet.
- **Pointer-event matrix** (`tests/e2e/notability-pointer-matrix.spec.ts`, 593 LOC). The Playwright pencil-gesture harness is genuine e2e coverage — exactly the kind of test TEST-AUDIT argues for. This is the model to extend to sync and AI flows.
- **Lazy-loaded route components** (`App.tsx:18–31`). Panels load only when visible; first paint does not pull in TipTap, Katex, Notability. Hard to get right, done correctly here.
- **Per-file service granularity.** `services/` has ~45 files, each with a single subject (`notesStorage`, `bookmarkStorage`, `readingHistory`). This is where R4 is respected.

## Questions the next session should answer

1. **Where does the `Provider` interface live — `services/aiProvider.ts`, a new `services/ai/Provider.ts`, or a shared `services/ai/` directory that both browser and edge function import?** (Edge is Deno, browser is Vite; the import path matters.) Recommendation lean: new `services/ai/providers.ts` + dedicated Deno-compatible module.
2. **Is the 5-way split in PLAN Phase 0 the right shape, or does the state reducer (recommendation 9) need to come first?** If the reducer lands first, the split's seams become obvious; if the split lands first, you may extract too eagerly and have to re-cut.
3. **Commit to deletion, not reorganization, for the 861 → 150 test cut?** TEST-AUDIT Section C lists candidates. The real question is whether PRs #N+1..N+12 actually delete files vs just renaming them.
4. **Fix egress in `syncService.ts` (Phase 2), or jump to Phase 3 Drive migration?** Phase 2 is ~2 days and buys runway; Phase 3 is 4 weeks and eliminates the problem. Phase 2 is the right call only if Phase 3 is at real risk of slipping past 2026-05-05 (the Supabase grace-period end).
5. **Does the R13 reviewer get upgraded to also check file-size budgets automatically?** Right now R4 is human-enforced; a 30-line hook in `review-prompt.md` could reject any diff that grows a file past 300 LOC without explicit `Acked-budget-growth:` attestation.
